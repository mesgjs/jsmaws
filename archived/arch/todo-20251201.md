# To-Do List (2025-12-01)

## Passing Request Details

- The operator should pass the routing details (e.g. applet, pool, method, routeParams, routeTail) and the *original* request URL to the responder
  - The operator should not pass, and responders should not expect, the path or query params separately, as these can be derived from the URL
- Responders should forward the original URL and routing data to the applet
- The applet can use the URL and URLSearchParams APIs to access elements of the URL or query (search) parameters
- Pass the route params and tail to the applet as `routeParams` and `routeTail` so their origin is unambiguous

## Timeouts

- Implement configurable responder "request", "idle", and "connection" timeouts
  - Request (`reqTimeout`): Begins when request is received and cancels when response is received (default 30s)
  - Idle (`idleTimeout`): Begins when there are no active requests and cancels when request is received (default 0 (none))
  - Connection (`conTimeout`): Begins when connection is created and cancels when connection is closed (default 0 (none))
  - Configuration hierarchy: global (lowest priority) -> pool -> route (highest priority)
  - The applet should receive all final timeout values when the request is initiated ("good citizen rulebook")
