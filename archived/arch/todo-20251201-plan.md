# Implementation Plan: Request Details and Timeouts

**Status**: Phase 1 COMPLETE | Phase 2 COMPLETE (Implementation) | Documentation and Testing Pending

**Last Updated**: 2025-12-02

## Overview

This document provides a detailed implementation plan for two related improvements:
1. Passing original request URL and routing details to applets
2. Implementing configurable timeout hierarchy (request, idle, connection)

## Part 1: Request URL and Routing Details

### Current State

**Problem**: The operator currently extracts `pathname` from the URL and never passes the query string. Applets receive empty `query` objects.

**Current Flow**:
```
Client Request: GET /api/users?filter=active&page=2
  ↓
Operator: new URL(req.url).pathname → "/api/users"
  ↓
IPC: { path: "/api/users", params: {...}, tail: "..." }
  ↓
Responder: query = fields.at('query') || new NANOS() → empty
  ↓
Applet: receives { path, params, query: {}, tail }
```

### Proposed Solution

**Pass the complete original URL** and let applets parse it using standard Web APIs.

**New Flow**:
```
Client Request: GET /api/users?filter=active&page=2
  ↓
Operator: req.url (complete URL)
  ↓
IPC: { url: "https://example.com/api/users?filter=active&page=2", 
       routeParams: {...}, routeTail: "..." }
  ↓
Responder: forwards url, routeParams, routeTail
  ↓
Applet: const url = new URL(request.url)
        const params = new URLSearchParams(url.search)
```

- Destructuring with rest parameters allows future fields to be added without modifying `createRequest()`

- Clear naming makes the origin of routing values unambiguous

## Part 2: Configurable Timeout Hierarchy

**Implementation Status**: COMPLETE ✅ (Code implementation done, documentation and tests pending)

### Current State

**Problem**: Only `reqTimeout` exists at pool level. No idle or connection timeouts. No route-level overrides. Additionally, the operator has hardcoded in-line timeout values in `setRequestHandler()` calls that need to be made configurable.

**Current Implementation**:

In [`src/responder-process.esm.js`](../src/responder-process.esm.js:169-179):
```javascript
const poolConfig = this.config.getPoolConfig(this.poolName);
const reqTimeout = poolConfig?.at('reqTimeout', 30) || 30;

const timeout = setTimeout(() => {
  if (this.activeRequests.has(id)) {
    console.warn(`[${this.processId}] Request ${id} timed out`);
    this.cleanupRequest(id);
    this.sendErrorResponse(id, 504, 'Gateway Timeout').catch(console.error);
  }
}, reqTimeout * 1000);
```

### Proposed Solution

Implement three timeout types with configuration hierarchy: global → pool → route.

### Timeout Types

#### 1. Request Timeout (`reqTimeout`)
- **Purpose**: Maximum time to process a single request
- **Starts**: When request is received by responder
- **Ends**: When response is sent (first frame for streaming)
- **Default**: 30 seconds
- **Applies to**: All request types (response, stream, bidi)

#### 2. Idle Timeout (`idleTimeout`)
- **Purpose**: Maximum time connection can be idle between frames
- **Starts**: After final frame is sent (when `final: true`)
- **Ends**: When next frame arrives
- **Default**: 0 (disabled)
- **Applies to**: Stream and bidi modes only
- **Semantics**: Only active between frames, NOT during request processing
  - Non-final chunks: Request timeout active, no idle timeout
  - Final chunk sent: Request timeout cleared, idle timeout starts
  - Next frame arrives: Idle timeout cleared, request timeout starts again
- **Note**: Each idle period gets the full configured timeout duration, regardless of previous request processing time

#### 3. Connection Timeout (`conTimeout`)
- **Purpose**: Maximum total lifetime of a connection
- **Starts**: When connection is established (first frame)
- **Ends**: When connection closes
- **Default**: 0 (disabled)
- **Applies to**: Stream and bidi modes only

### Configuration Hierarchy

**Priority**: Route > Pool > Global

```javascript
/**
 * Get timeout configuration with hierarchy: route > pool > global
 * @param {string} poolName Pool name
 * @param {Object} routeSpec Route specification (optional)
 * @returns {Object} Timeout configuration
 */
getTimeoutConfig (poolName, routeSpec = null) {
  // Global defaults
  const defaults = {
    reqTimeout: this.config.at('reqTimeout', 30),
    idleTimeout: this.config.at('idleTimeout', 0),
    conTimeout: this.config.at('conTimeout', 0),
  };
  
  // Pool overrides
  const poolConfig = this.getPoolConfig(poolName);
  const poolTimeouts = {
    reqTimeout: poolConfig?.at('reqTimeout', defaults.reqTimeout),
    idleTimeout: poolConfig?.at('idleTimeout', defaults.idleTimeout),
    conTimeout: poolConfig?.at('conTimeout', defaults.conTimeout),
  };
  
  // Route overrides (highest priority)
  if (routeSpec) {
    return {
      reqTimeout: routeSpec.at('reqTimeout', poolTimeouts.reqTimeout),
      idleTimeout: routeSpec.at('idleTimeout', poolTimeouts.idleTimeout),
      conTimeout: routeSpec.at('conTimeout', poolTimeouts.conTimeout),
    };
  }
  
  return poolTimeouts;
}
```

#### Step 4: Implement Idle Timeout

**Add to Responder Process**:
```javascript
/**
 * Start idle timeout for streaming/bidi connections
 */
startIdleTimeout (id, idleTimeout) {
  if (idleTimeout <= 0) return null;  // Disabled
  
  return setTimeout(() => {
    const requestInfo = this.activeRequests.get(id);
    if (requestInfo && requestInfo.keepAlive) {
      console.warn(`[${this.processId}] Connection ${id} idle timeout after ${idleTimeout}s`);
      this.cleanupRequest(id);
      this.sendErrorResponse(id, 408, 'Request Timeout').catch(console.error);
    }
  }, idleTimeout * 1000);
}

/**
 * Reset idle timeout (called on message activity)
 */
resetIdleTimeout (id, idleTimeout) {
  const requestInfo = this.activeRequests.get(id);
  if (!requestInfo) return;
  
  // Clear existing idle timeout
  if (requestInfo.idleTimeout) {
    clearTimeout(requestInfo.idleTimeout);
  }
  
  // Start new idle timeout
  requestInfo.idleTimeout = this.startIdleTimeout(id, idleTimeout);
}
```

## Key Architectural Decisions

### Idle Timeout Semantics (Phase 2)
After discussion and clarification, the idle timeout behavior was defined as:
- **Active only between frames**, not during request processing
- **Starts** after a final frame is sent (`final: true`)
- **Cleared** when a new frame arrives (processing begins)
- **Full duration** for each idle period (not shortened by previous processing time)

This ensures that:
1. Request timeout handles processing duration
2. Idle timeout handles gaps between requests/frames
3. Connection timeout handles total connection lifetime
4. No overlap or confusion between timeout types

### Timeout Disabling (Phase 2)
All three timeout types can be disabled by setting them to 0:
- `reqTimeout=0` - No request timeout (use with caution)
- `idleTimeout=0` - No idle timeout (common for client-driven connections)
- `conTimeout=0` - No connection lifetime limit (common for persistent connections)

### Implementation Notes (Phase 2)
1. **Operator timeout resolution**: Captures `routeSpec` in closure scope and passes as parameter to prevent race conditions
2. **Race condition fix**: `routeSpec` passed through all async handlers (`handleFrameResponse`, `handleResponseStream`, `handleBidiUpgrade`, `readBidiFrames`) to prevent concurrent request overwrites
3. **Responder timeout storage**: Stores resolved timeouts in `requestInfo.timeouts` for access throughout request lifecycle
4. **Cleanup**: All three timeout handles cleared in `cleanupRequest()`
5. **User edit**: Added null-check for `reqTimeout` to handle 0=disabled case
